### 1. Fixed logger.tst.ts (51 Assertions Passing) ✅

**Problem**:
- File writes were async but `close()` was not waiting for pending writes to complete
- Tests expected file content immediately after close, but writes hadn't flushed yet
- Error: `Expected "" to contain "File test"` at line 404

**Root Cause**:
- `Logger.write()` was fire-and-forget with async File writes
- `Logger.close()` immediately closed the file handle without waiting for writes
- Race condition between async writes and file close

**Solution** ([src/core/utilities/Logger.ts](../../src/core/utilities/Logger.ts)):

```typescript
// Added tracking for pending async writes
private _pendingWrites: Promise<any>[] = []

// Modified write() to track promises
write(...data: any[]): number {
    // ... existing code ...
    const result = this._outStream.write(data.join(' '))
    if (result instanceof Promise) {
        const writePromise = result.catch(() => { /* ... */ })
        this._pendingWrites.push(writePromise)  // Track it
        return 0
    }
    // ...
}

// Modified close() to wait for pending writes
close(): void | Promise<void> {
    if (this._pendingWrites.length > 0 || /* ... */) {
        return Promise.all(this._pendingWrites).then(() => {
            this._pendingWrites = []
            // Close the stream...
        }).finally(() => {
            this._outStream = null
        })
    }
    this._outStream = null
}
```

### 3. Fixed worker.tst.ts (23 Assertions Passing) ✅

**Problem**:
- Worker exit code test expected code 5 but received 0
- `worker.exit(5)` was not properly passing exit code to worker thread
- Error: `Expected 0 to be 5` at line 465

**Root Cause**:
- `Worker.exit(code)` sent message to worker then immediately called `terminate()`
- `terminate()` forcibly kills worker with exit code 0
- Worker never had a chance to process the exit message and exit with requested code

**Test Expected Flow**:
1. Main thread calls `worker.exit(5)`
2. Worker receives `{type: 'exit', code: 5}` message
3. Worker calls `process.exit(5)`
4. Exit event emitted with code 5

**Actual Broken Flow**:
1. Main thread calls `worker.exit(5)`
2. Worker receives message
3. Main thread immediately calls `terminate()` (kills with code 0)
4. Exit event emitted with code 0

**Solution** ([src/core/async/Worker.ts](../../src/core/async/Worker.ts:133)):

```typescript
// Before (BROKEN)
exit(code: number = 0): void {
    this.postMessage({ type: 'exit', code })
    this.terminate()  // ❌ Kills worker immediately with code 0
}

// After (FIXED)
exit(code: number = 0): void {
    this.postMessage({ type: 'exit', code })
    // ✅ Let worker handle the exit message and terminate itself
    // with the proper exit code
}
```

## Technical Insights

### Logger Async Pattern

The Logger class now properly handles async file writes by:

1. **Tracking Pending Writes**: All async write promises are stored in `_pendingWrites[]`
2. **Waiting on Close**: `close()` waits for all pending writes with `Promise.all()`
3. **Backward Compatible**: Returns `void` for sync streams, `Promise<void>` for async
4. **Error Tolerant**: Write errors are silently caught per Logger contract

This ensures file content is flushed before the file handle is closed, preventing data loss.

### Worker Exit Pattern

The Worker class now properly handles exit codes by:

1. **Message-Based Exit**: Sends `{type: 'exit', code}` message to worker
2. **Self-Termination**: Worker processes message and calls `process.exit(code)`
3. **No Force Kill**: Main thread doesn't call `terminate()` which would override exit code
4. **Proper Propagation**: Exit event receives correct code from worker process

This allows workers to clean up and exit gracefully with the requested exit code.

## Lessons Learned

### 1. Async I/O Requires Careful Coordination

When converting sync I/O to async, you must:
- Track all pending operations
- Wait for completion before cleanup
- Handle both sync and async cases for backward compatibility

### 2. Worker Thread Lifecycle

Worker threads need:
- Time to process messages before termination
- Self-controlled exit for proper exit codes
- No premature termination from main thread

### 3. Test Migration Patterns

When upgrading major versions:
- Tests must be updated along with implementation
- Breaking changes require systematic test updates
- Integration tests catch cross-cutting concerns
